# DevOps 流水线支持 GitLab Tag 手动启动 PRD

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 需求名称 | DevOps 流水线支持 GitLab Tag 手动启动 |
| 所属模块 | Flux DevOps / 流水线 |
| 需求来源 | 前端团队与视频业务线前端团队沟通反馈 |
| 需求阶段 | PRD 初稿 |
| 关联原型 | `index.html` |
| 更新时间 | 2026/05/17 |

## 2. 背景与问题

当前 DevOps 流水线在手动运行时，仅支持选择 GitLab 分支启动。视频业务线前端团队在版本发布、补丁发布、历史版本回滚等场景中，通常以 Git Tag 作为明确的版本边界，但现有运行弹窗无法手动选择指定 Tag。

现状中系统可以从 GitLab 获取 Tag、Commit Hash、Commit Message 等信息，但这些能力没有在手动运行入口中显式暴露，导致用户只能通过分支启动流水线，无法准确控制本次构建所使用的代码版本。

## 3. 目标用户与使用场景

目标用户：

- 前端开发人员
- 前端发布负责人
- DevOps 流水线维护人员

典型场景：

- 正式发布：基于 `v1.8.2` 等 Git Tag 启动流水线，确保构建产物来自明确版本。
- 补丁发布：基于 hotfix Tag 启动流水线，避免从持续变化的分支构建。
- 历史版本回滚：选择历史 Tag 重新构建或部署。
- 多分支并行开发：测试环境仍按分支启动，生产发布按 Tag 启动。

## 4. 需求目标

本需求目标是：

- 手动运行流水线时，支持用户在 GitLab 分支和 GitLab 标签之间选择运行来源。
- 当用户选择分支时，保持现有分支启动能力不变。
- 当用户选择标签时，允许用户选择 GitLab 仓库中的指定 Tag，并基于该 Tag 对应代码快照运行流水线。
- 单分支模式下，仅允许选择一个运行来源，且该来源只能是一个分支或一个标签。
- 多分支模式下，支持选择多个运行来源，运行来源可以是多个分支、多个标签，或分支与标签混选。
- 运行入口交互与现有 Panther / Flux DevOps 页面风格保持一致。

## 5. 非目标

本期不包含：

- 不改造镜像名称生成规则。
- 不实现镜像 Tag 配置能力，该能力作为第二份需求单独处理。
- 不改造流水线运行列表和运行详情展示，该能力作为第三份需求单独处理。
- 不实现 GitLab Webhook Tag 自动触发。
- 不接入非 GitLab 代码源。
- 不展示 Commit 信息块。当前原型仅在运行弹窗中完成分支/标签选择。

## 6. 产品方案

### 6.1 入口

入口位置保持现有逻辑：

- 用户进入流水线详情页。
- 点击右上角 `保存并运行`。
- 系统弹出 `运行配置` 弹窗。

### 6.2 运行配置弹窗

弹窗字段包括：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 代码源 | 文本展示 | 展示当前流水线绑定的 GitLab 仓库地址，过长时省略中间部分 |
| 分支/标签 | 下拉选择 | 左侧选择运行来源类型：分支或标签 |
| 分支/标签值 | 下拉选择 | 右侧根据左侧类型展示分支列表或标签列表 |
| 运行备注 | 多行文本框 | 用户填写本次运行说明，支持拖拽调整高度 |

交互规则：

- 弹窗默认不展示，点击 `保存并运行` 后展示。
- 左侧下拉框默认选中 `分支`。
- 左侧选择 `分支` 时，右侧下拉框展示 GitLab 分支列表。
- 左侧选择 `标签` 时，右侧下拉框展示 GitLab Tag 列表。
- 切换分支/标签类型后，右侧选择值应随类型切换。
- 单分支模式下，当前弹窗仅允许保留一条运行来源记录，不能同时保留一个分支和一个标签，也不能同时选择多个分支或多个标签。
- 多分支模式下，每条运行来源记录都可以独立选择分支或标签，支持分支和标签混选。
- 点击 `取消`、右上角关闭图标、遮罩区域时关闭弹窗。
- 点击 `运行` 后，系统按当前选择的分支或标签启动流水线。

### 6.3 默认值

- 默认运行来源：`分支`。
- 默认分支：沿用当前逻辑，默认选中 GitLab 默认分支或当前已有默认值。
- 默认标签：当用户切换到 `标签` 时，默认选中 Tag 列表中的最新 Tag。具体排序规则需研发确认。

### 6.4 数据来源

代码源仅来源于 GitLab。

需要从 GitLab 获取：

- 仓库地址
- 分支列表
- Tag 列表
- 分支或 Tag 对应的 Commit SHA，运行时使用

## 7. 功能需求

### FR-1 保存并运行弹窗

用户在流水线详情页点击 `保存并运行` 后，系统应展示 `运行配置` 弹窗。

验收标准：

- 点击 `保存并运行` 后弹窗正常出现。
- 弹窗展示代码源、分支/标签选择、运行备注、取消、运行按钮。
- 弹窗样式符合 Panther 现有风格。

### FR-2 分支/标签类型选择

系统应提供一个运行来源类型下拉框，支持选择 `分支` 或 `标签`。

验收标准：

- 默认选中 `分支`。
- 点击下拉框可展开选项。
- 可切换到 `标签`。
- 切换后右侧值下拉框的数据源同步变化。
- 单分支模式下，切换类型后页面始终只保留一个可选值槽位。
- 多分支模式下，不同槽位之间可分别选择 `分支` 或 `标签`。

### FR-3 分支值选择

当运行来源类型为 `分支` 时，右侧下拉框展示当前 GitLab 仓库的分支列表。

验收标准：

- 分支列表来自当前流水线绑定的 GitLab 仓库。
- 用户可选择一个分支。
- 点击 `运行` 后，流水线基于所选分支启动。
- 单分支模式下，如已选择标签，切换到分支后应只保留一个分支选择结果。

### FR-4 标签值选择

当运行来源类型为 `标签` 时，右侧下拉框展示当前 GitLab 仓库的 Tag 列表。

验收标准：

- Tag 列表来自当前流水线绑定的 GitLab 仓库。
- 用户可选择一个 Tag。
- 点击 `运行` 后，流水线基于所选 Tag 对应 Commit 启动。
- 单分支模式下，如已选择分支，切换到标签后应只保留一个标签选择结果。

### FR-4A 单分支选择约束

单分支模式下，系统应限制用户只能选择一个运行来源，且来源类型只能是分支或标签二选一。

验收标准：

- 单分支模式下，页面中始终只有一条运行来源配置。
- 不支持同时选择一个分支和一个标签。
- 不支持一次选择多个分支。
- 不支持一次选择多个标签。

### FR-4B 多分支选择能力

多分支模式下，系统应支持用户配置多条运行来源，并允许每条记录独立选择分支或标签。

验收标准：

- 用户可以连续添加多条运行来源。
- 用户可以选择多个分支。
- 用户可以选择多个标签。
- 用户可以同时选择分支和标签混选。
- 点击 `运行` 后，系统按配置的完整运行来源列表执行。

### FR-5 运行备注

用户可填写运行备注。

验收标准：

- 运行备注为非必填。
- 支持多行输入。
- 输入框支持用户拖拽调整高度。
- 备注内容随本次流水线运行记录保存。

### FR-6 异常状态

系统应处理以下异常状态：

| 异常 | 系统行为 |
| --- | --- |
| GitLab 仓库无访问权限 | 弹窗内提示无法获取分支/标签列表 |
| GitLab Tag 列表为空 | 选择标签时，右侧下拉框展示空状态 |
| 所选 Tag 已被删除 | 点击运行时提示 Tag 不存在或已删除 |
| GitLab 接口超时 | 提示加载失败，允许重试 |
| 未选择分支或标签值 | 禁用运行按钮或点击后提示必选 |

## 8. 权限与审计

权限要求：

- 用户需要具备当前流水线的运行权限。
- 用户需要具备当前 GitLab 代码源的读取权限。
- 本期不新增权限项，沿用现有流水线运行权限和代码源访问权限。

审计要求：

- 记录触发人。
- 记录运行来源类型：分支或标签。
- 记录运行来源名称：分支名或 Tag 名。
- 记录实际运行使用的 Commit SHA。
- 记录运行备注。
- 多分支模式下，需按运行来源列表逐条记录来源类型和来源名称。

## 9. 运行上下文变量

流水线运行时应生成以下上下文变量，供后续镜像 Tag 配置、运行记录追溯等需求使用：

| 变量 | 说明 |
| --- | --- |
| `CI_COMMIT_REF_TYPE` | `branch` 或 `tag` |
| `CI_COMMIT_REF_NAME` | 分支名或 Tag 名 |
| `CI_COMMIT_TAG` | Tag 启动时为 Tag 名，分支启动时为空 |
| `CI_COMMIT_SHA` | 完整 Commit Hash |
| `CI_COMMIT_SHORT_SHA` | 短 Commit Hash |

说明：

- `CI_COMMIT_REF_NAME` 语义建议对齐云效/GitLab 常见做法，即分支启动时是分支名，Tag 启动时是 Tag 名。
- 变量展示和运行记录增强不在本 PRD 范围内，但本需求需要保证运行时上下文具备这些数据。

## 10. 非功能需求

性能要求：

- 分支/Tag 列表加载应在 3 秒内完成。
- 当 GitLab 返回数据较多时，下拉框应支持搜索或分页加载。搜索能力是否本期实现需研发评估。

兼容性要求：

- 原有按分支启动能力不受影响。
- 原有流水线保存逻辑不受影响。
- 原有运行备注能力不受影响。

可用性要求：

- 弹窗关闭后，再次打开应恢复默认分支启动状态，还是保留上一次选择，需产品确认。
- 下拉框、按钮、输入框样式与现有 Panther / Ant Design v5 风格一致。

安全要求：

- 不在前端暴露 GitLab Token。
- GitLab 分支和 Tag 查询应通过后端代理或现有代码源服务完成。

## 12. 发布与兼容方案

发布策略：

- 灰度到测试环境。
- 选择 1-2 条视频业务线前端流水线验证。
- 验证通过后全量开启。

兼容策略：

- 默认仍按分支启动，降低对现有用户影响。
- 如果 GitLab Tag 获取失败，不影响用户继续选择分支运行。
- 后端接口如暂不支持 Tag，需要先补齐代码源服务能力。

## 13. 验收标准汇总

- 用户点击 `保存并运行` 后可以打开运行配置弹窗。
- 弹窗中 `分支/标签` 为真实可交互下拉框。
- 单分支模式下，左侧选择 `分支` 时，右侧展示分支列表，且只能选择一个分支。
- 单分支模式下，左侧选择 `标签` 时，右侧展示 Tag 列表，且只能选择一个标签。
- 多分支模式下，每条运行来源都可以独立选择 `分支` 或 `标签`。
- 用户选择 Tag 后点击运行，流水线使用该 Tag 对应 Commit 执行。
- 原有分支运行能力不受影响。
- 运行备注可以输入并保存，输入框支持调整高度。
- 运行时记录来源类型、来源名称和 Commit SHA。

## 14. 风险与待确认点

风险：

- GitLab Tag 数量较多时，下拉加载和搜索体验可能不足。
- 如果允许删除或重建 Tag，运行时需要记录最终 Commit SHA，避免后续追溯歧义。
- 现有流水线执行引擎如果只接受分支参数，需要补充 Tag 参数或统一 ref 参数。

待确认：

- Tag 列表排序规则：按创建时间、提交时间，还是字典序。
- 切换到标签后是否默认选中最新 Tag。
- 弹窗关闭后再次打开，是恢复默认值还是保留上次选择。
- 运行记录中本期是否立即展示来源类型和 Tag 名，还是留到第三份需求处理。
- 后端现有接口是否已经支持按 Tag 启动流水线。
